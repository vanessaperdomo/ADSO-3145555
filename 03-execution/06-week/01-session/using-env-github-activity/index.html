Estructura esperada

¿Qué debe explicar este landing page?

Este landing presenta el propósito del repositorio, la política de ambientes, la regla de no tocar ramas padre, el manejo de cambios mediante HU y el patrón correcto de ramas hijas por ambiente.

1. Contexto del repositorio

El repositorio trabaja con ambientes separados y una estrategia disciplinada de promoción de cambios.

  • Repo: using-env-github-activity
  • Ambientes: develop, qa, main
  • Cambios planificados por HU

2. Regla central

Las ramas padre nunca se modifican directamente. Todo cambio debe entrar por una rama hija controlada.

  • No commits directos a ramas padre
  • No Merge Requests cruzadas entre padres
  • Solo ramas hijas HU-env → rama padre correspondiente

3. Evidencia operativa

Cada ambiente recibe una HU derivada de su propia rama padre, respetando el alcance planificado.

  • HU-01-develop → develop
  • HU-01-qa → qa
  • HU-01-main → main

Reglas del flujo

Política de trabajo por ambientes

develop

Desarrollo

Ambiente de construcción y ajuste. Aquí pueden existir errores, cambios frecuentes y validaciones iniciales.

qa

Calidad

Recibe cambios cuando el desarrollo de la HU ya culminó y se requiere validar el comportamiento esperado.

main

Producción

Únicamente recibe cambios probados. Debe mantenerse estable y libre de trabajo en curso no verificado.

Reglas obligatorias

Se debe hacer

  • Planificar cambios mediante HU.
  • Derivar cada HU desde la rama padre correcta.
  • Crear Merge Request solo de la rama hija a su padre directo.
  • Conservar la trazabilidad por ambiente.

No se debe hacer

  • No tocar directamente develop, qa o main.
  • No mezclar cambios entre ambientes padre.
  • No abrir MR de develop hacia qa o de qa hacia main.
  • No usar una sola rama hija para todos los ambientes.

Flujo HU

Patrón correcto de derivación por ambiente

Una HU no modifica ramas padre de forma directa. Se crean ramas hijas por ambiente y cada una abre su propia MR únicamente contra su rama padre correspondiente.

Ejemplo de alcance de una HU

Si la HU-01 tiene como propósito crear la base de datos test-security, entonces el alcance de la HU se limita al DDL necesario para ese objetivo. No se deben mezclar cambios ajenos al alcance definido. 

-- Eliminar base de datos
DROP DATABASE IF EXISTS test-security;

-- Usar base de datos
USE test-security;

-- Crear base de datos
CREATE DATABASE test-security;

Comandos

Secuencia por ambiente

Use los botones para cambiar entre el flujo de comandos de cada ambiente. Cada bloque representa una rama hija distinta.

HU-01 para develop

git switch develop
git switch -c HU-01-develop
git pull origin HU-01-develop
# Crear MR de HU-01-develop hacia develop

HU-01 para qa

git switch qa
git switch -c HU-01-qa
git pull origin HU-01-qa
# Crear MR de HU-01-qa hacia qa

HU-01 para main

git switch main
git switch -c HU-01-main
git pull origin HU-01-main
# Crear MR de HU-01-main hacia main

Control de errores

Qué se debe evitar

Antipatrón 1

Commit directo a ramas padre

Rompe la trazabilidad, omite revisión formal y aumenta el riesgo de inconsistencia entre ambientes.

Antipatrón 2

Cruzar Merge Requests entre ramas padre

Genera contaminación entre ambientes. La promoción correcta es rehacer la HU en su rama hija respectiva por ambiente.

Antipatrón 3

Una sola rama hija para todo

Pierde aislamiento por ambiente y dificulta la validación disciplinada de cada promoción.

Antipatrón 4

HU sin alcance claro

Una HU debe tener propósito definido. No se deben mezclar cambios funcionales, estructurales o correctivos no relacionados.

Conclusión operativa

La regla es simple

Las ramas padre no se tocan. Cada cambio nace como una HU, se deriva a una rama hija específica por ambiente, y solo entra a develop, qa o main mediante su Merge Request correspondiente.
